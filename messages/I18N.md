# i18n Translation Workflow

## Structure

```
messages/
  en/          # English translations (21 namespaces)
  zh-CN/       # Simplified Chinese translations
  normalize.ts # Converts dotted keys to nested objects for next-intl
```

Each namespace (e.g. `auth`, `dashboard`, `agents`) has a corresponding JSON file in both locale directories.

## Audit: Find Missing Translations

```bash
pnpm i18n:audit
```

Scans all source files for `useTranslations()` / `t()` calls and cross-references against both locale message files. Reports:
- **Missing keys** — used in code but absent from one or both locales
- **Orphan keys** — present in message files but never referenced in code

Flags: `--json` for machine-readable output, `--dir <path>` to override scan scope.

Exit code 1 if any missing keys are found (CI-friendly).

## Fill: Auto-Translate Missing Keys

Inside a Claude Code session, run:

```
/i18n-fill
```

This reads the audit report, translates missing keys using the agent's built-in multilingual capability, and writes them back to the correct JSON files. No API key or external dependency needed.

## Adding New Translations

1. Add the key to your component: `t("mySection.myKey")`
2. Add the English value to `messages/en/{namespace}.json`
3. Add the Chinese value to `messages/zh-CN/{namespace}.json`
4. Run `pnpm i18n:audit` to verify nothing is missing

Use **dotted-key format** (e.g. `"login.title": "Sign in"`), not nested objects. The `normalize.ts` module handles conversion to nested format for next-intl at runtime.
