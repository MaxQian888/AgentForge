## Context

当前 `app/(dashboard)/project/page.tsx` 只提供单一 `Board` 视图，依赖 `components/kanban/*` 和 `lib/stores/task-store.ts` 展示任务状态列，并通过现有 `transitionTask` 完成状态拖拽。这个实现已经覆盖了最小 Kanban 流转，但它离 PRD 的 P0“多视图（Board/List/Timeline/Calendar），支持拖拽”还有明显缺口：

- 项目任务页没有视图切换容器，用户只能按状态列查看任务。
- `task-store` 和前后端任务合同目前只有状态、优先级、指派、成本、创建/更新时间等字段，没有任务级计划起止日期，无法支撑 Timeline / Calendar 的稳定布局。
- Go 后端和数据库已有 `sprint_id`、`blocked_by` 等项目规划相关字段，但这些信息只能表达任务所属 Sprint 或依赖关系，不能替代单个任务在时间轴/日历里的排期窗口。
- 现有 Board 拖拽已经是用户熟悉的交互，本次设计需要在保留该能力的前提下，把 List、Timeline、Calendar 纳入同一任务工作区，而不是拆成互相割裂的新页面。

这意味着本次变更既涉及前端任务工作区结构，也涉及任务数据模型和 API 契约，是一个值得先写清楚技术边界的跨层 change。

## Goals / Non-Goals

**Goals:**
- 在项目任务页提供 `Board / List / Timeline / Calendar` 四种视图切换，并保持同一项目上下文、同一任务集合和同一筛选条件。
- 保留现有 Board 状态拖拽，并为 Timeline / Calendar 增加基于任务计划日期的拖拽排期能力。
- 为任务补齐支撑多视图的最小计划字段，让前后端对“已排期 / 未排期”有统一表达。
- 为不同视图定义一致的空状态、筛选和详情入口，避免用户切换视图后丢失上下文。

**Non-Goals:**
- 不在本次 change 中实现完整 Sprint 规划中心、燃尽图或依赖图。
- 不扩展跨项目日历、组织级资源管理或高级排班算法。
- 不重做任务详情面板的信息架构，只要求它在多视图下继续可用。
- 不把 List 视图扩展成可配置列、批量编辑或无限滚动数据表产品。

## Decisions

### Decision: 在现有项目页内构建共享的任务工作区容器，而不是拆成四个独立页面

项目任务管理继续留在 `app/(dashboard)/project/page.tsx`，在页头新增视图切换与共享筛选栏，再由具体视图组件渲染 Board、List、Timeline 或 Calendar。这样可以复用现有项目选择逻辑、任务详情侧板和任务创建入口，也能保证用户切换视图时不丢失当前项目和筛选条件。

备选方案是把四种视图拆成独立路由，例如 `/project/board`、`/project/calendar`。这样路由更直观，但会重复项目加载、筛选同步和详情导航逻辑，还会让“一个项目任务工作区”被切碎，因此不采用。

### Decision: 为任务引入显式的计划日期字段，而不是复用 `metadata` 或仅依赖 `sprint_id`

Timeline / Calendar 需要可排序、可验证、可拖拽回写的稳定日期语义。本次设计采用任务级显式计划字段，例如 `plannedStartDate` / `plannedEndDate`（后端可落到 nullable date/date-time 字段），并让前后端 DTO、store 和更新接口统一承载它们。Sprint 仍然表示任务所在迭代，但不直接等价于任务的具体排期窗口。

备选方案一是把计划信息塞进 `tasks.metadata`。这个做法迁移成本较低，但会让查询、排序、校验和前后端类型变得松散，不适合作为多视图交互的主合同。备选方案二是只使用 Sprint 起止日期推导任务位置；这样无法表达同 Sprint 内多个任务的细粒度时序，也无法处理未进 Sprint 但已排期的任务，因此不采用。

### Decision: 将拖拽能力按视图语义分层，而不是强求所有视图使用同一种拖拽模型

Board 继续使用状态列拖拽，拖动结果更新任务状态；Timeline / Calendar 使用排期拖拽，拖动结果更新计划日期；List 视图保留为高密度浏览和快速编辑入口，不引入没有明确排序语义的“任意行拖拽”。这样既满足“多视图 + 支持拖拽”的核心目标，也避免为了 List 强行引入一个当前数据模型并不支持的排序字段。

备选方案是为四种视图都添加统一拖拽协议，例如 List 行重排和 Calendar 卡片换列都映射到一个“rank”字段。这会显著扩大数据模型和交互复杂度，也超出用户当前明确需求，因此不采用。

### Decision: 为未排期任务提供显式的 Unscheduled 表达，而不是在 Timeline / Calendar 中隐藏它们

不是所有任务在创建时都会立即拥有计划日期。为了避免任务在切到 Timeline / Calendar 后“消失”，这两个视图都需要一个明确的未排期区域或提示，并允许用户从未排期状态进入排期交互。这样既能保证多视图看到的是同一任务集合，也能把“需要安排时间”的工作暴露出来。

备选方案是自动使用 `createdAt` 或当前日期填充缺失排期。这会制造虚假的计划数据，并影响后续真实调度，因此不采用。

## Risks / Trade-offs

- [任务计划字段会触及数据库、Go DTO 和前端 store 多层契约] → 采用 nullable 字段与向后兼容读取，未排期任务继续可被旧数据链路消费。
- [不同视图可能出现状态不同步或重复转换逻辑] → 将任务规范化和更新映射集中在共享 store / view model，而不是分散在各视图组件里。
- [Timeline / Calendar 拖拽的交互复杂度高于现有 Board] → 第一版仅支持单任务拖拽改期和基础日期窗口展示，不在本次加入缩放、批量拖拽或复杂依赖联动。
- [List 不支持自由拖拽可能与“支持拖拽”的直觉存在落差] → 在规格中明确拖拽适用于 Board、Timeline、Calendar 等有明确空间语义的视图，List 承担高密度管理与快速编辑角色。

## Migration Plan

1. 先补齐 `task-multi-view-board` capability spec，锁定视图切换、共享筛选、拖拽语义和未排期行为。
2. 在实现阶段扩展任务数据合同，引入最小计划日期字段，并保持现有任务记录在字段为空时仍可正常读取。
3. 将项目任务页重构为共享任务工作区，默认仍以 Board 作为初始视图，再逐步接入 List、Timeline、Calendar。
4. 如果多视图上线后发现排期字段或新视图稳定性不足，可先仅保留 Board 入口并隐藏新视图切换，同时不回滚已有兼容字段。

## Open Questions

- Calendar 第一版是否只需要月视图，还是要同时提供周视图切换；当前设计默认先满足单一主日历视图即可。
- Timeline 是否需要按 assignee、status 或 sprint 分组切换；本次先要求它能表达时间范围与基础分组，不强制一次性支持全部维度。
