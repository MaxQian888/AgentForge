## Context

AgentForge 已经通过 [docs/part/PLUGIN_SYSTEM_DESIGN.md](d:\Project\AgentForge\docs\part\PLUGIN_SYSTEM_DESIGN.md)、[docs/part/PLUGIN_RESEARCH_TECH.md](d:\Project\AgentForge\docs\part\PLUGIN_RESEARCH_TECH.md) 和 [docs/part/PLUGIN_RESEARCH_PLATFORMS.md](d:\Project\AgentForge\docs\part\PLUGIN_RESEARCH_PLATFORMS.md) 明确了插件化方向，但当前仓库还没有一个可以指导实现的最小落地契约。现有代码面已经具备两个天然宿主：Go Orchestrator 负责服务、配置、生命周期与状态管理，TS Bridge 负责 Agent 运行时与 MCP 工具接入，但插件 manifest、运行时归属、状态同步和注册中心真相源都尚未固定。

这个变更要解决的不是“完整插件生态”，而是先落地第一阶段底座：让平台能识别插件、知道该由谁启动、知道当前是否可用，并把这些状态统一沉淀到一个注册中心。没有这层契约，后续工具插件、集成插件、审查插件和角色插件都会各自引入私有配置和状态模型。

## Goals / Non-Goals

**Goals:**
- 定义统一的插件 manifest 契约，覆盖插件身份、类型、运行时、权限声明和来源信息。
- 明确第一阶段可执行插件的运行时归属：Go 侧负责集成类子进程插件，TS Bridge 负责 MCP 工具类插件。
- 定义插件生命周期状态机和健康状态上报模型，确保 Go 与 TS 对插件状态使用同一套语义。
- 规定插件注册中心的权威职责，包括元数据存储、启停状态、来源、版本和运行时状态同步。
- 为后续角色、工作流、审查、市场与 SDK 扩展预留兼容字段和状态模型，而不在本次直接实现这些高层能力。

**Non-Goals:**
- 不实现远程插件市场、签名校验、审核流和评分评论体系。
- 不实现 WASM 运行时、Serverless 运行时或跨实例分布式插件编排。
- 不在本次内交付所有插件类型的完整执行能力；第一阶段只要求把运行时契约和注册中心边界落稳。
- 不定义完整前端插件市场 UI，只要求后端和桥接层可支撑后续管理界面接入。
- 不把插件 SDK、脚手架和开发者门户纳入本次变更范围。

## Decisions

### 1. 使用统一 manifest 作为所有插件的声明入口

平台将使用统一的 manifest 顶层结构来描述插件，而不是让不同插件类型各自维护不同配置格式。manifest 至少包含 `apiVersion`、`kind`、`metadata`、`spec`、`permissions` 和 `source` 信息，并允许不同 `kind` 在 `spec` 下扩展自己的运行时字段。

选择这个方向，是因为仓库现有设计已经反复强调 “声明式优先” 和 “双侧一致”。统一 manifest 能让安装、校验、注册、状态同步和未来签名验证都复用一套入口。

备选方案：
- 为每种插件类型维护独立配置格式。问题是会造成安装和校验逻辑分叉。
- 直接在代码中硬编码插件配置。问题是无法支撑市场、外部插件和版本管理。

### 2. 第一阶段采用双宿主运行时，而不是强行统一到单一插件框架

第一阶段运行时归属明确为：
- `ToolPlugin` 走 TS Bridge，使用 MCP 协议和现有 `src-bridge` 运行时能力。
- `IntegrationPlugin` 走 Go Orchestrator 子进程运行时，由 Go 侧负责启停、健康检查和事件回传。

这样做的原因是当前仓库的技术边界已经很清晰：TS Bridge 天然适合 MCP 工具与 Agent 侧能力，Go 天然适合守护式集成、配置和长期状态管理。统一到单一 runtime 会引入额外适配层，反而把 MVP 复杂化。

备选方案：
- 全部插件都放进 TS Bridge。问题是会把集成类插件和平台配置能力过度绑定到 TS 侧。
- 全部插件都放进 Go。问题是会削弱 MCP 原生兼容和现有 bridge 工具能力。

### 3. 插件注册中心由 Go Orchestrator 持有权威状态

插件注册中心的权威真相源放在 Go 侧。Go 负责保存插件元数据、启停状态、来源、版本、权限声明、最近健康时间和错误摘要；TS 侧只负责管理自己宿主下的运行态，并通过桥接事件把状态回报给 Go。

这样可以保持平台级状态只有一个来源，避免前端、Go、TS 三处各自缓存导致状态漂移。也更符合仓库里现有“Go 管业务真相、TS 管 AI/runtime”的结构。

备选方案：
- TS 持有工具插件真相，Go 只查询。问题是前端和服务侧聚合会依赖双来源状态。
- 前端直接持有插件管理状态。问题是刷新、重连和后台任务都会丢失权威记录。

### 4. 生命周期状态机采用统一语义，并允许宿主分别驱动

无论插件由哪一侧宿主管理，平台都统一使用以下状态语义：`installed`、`enabled`、`activating`、`active`、`degraded`、`disabled`。同时记录 `lastHealthAt`、`lastError`、`restartCount` 等运行时字段。

这样做可以让前端、任务编排和后续审查流程使用统一条件判断，而不需要理解“TS MCP server 的 active”和“Go plugin 进程存活”的不同细节。

备选方案：
- 每个宿主自定义状态枚举。问题是注册中心无法稳定聚合，也不利于后续告警和运维。

### 5. 第一阶段只支持本地来源与内置来源的安装模型

为了保持范围收敛，注册中心第一阶段仅要求支持：
- 仓库内置插件发现
- 本地路径安装/注册

远程 registry、签名验证和在线更新只在数据模型上留字段，不纳入本次实现要求。这样可以先把运行时和状态同步闭环打通，再扩到分发。

备选方案：
- 一开始就建设远程市场。问题是范围过大，且会把注意力从运行时契约转移到分发基础设施。

## Risks / Trade-offs

- [Risk] 双宿主运行时会让插件系统看起来不够“统一” -> Mitigation: 用统一 manifest、统一状态机和统一注册中心屏蔽宿主差异。
- [Risk] 注册中心由 Go 持有真相，可能增加 TS 状态同步复杂度 -> Mitigation: 明确 TS 只负责事件上报，不负责权威持久化。
- [Risk] 第一阶段仅支持本地/内置来源，未来远程分发可能需要迁移 -> Mitigation: 提前在 manifest/source 和 registry 记录中保留远程来源字段。
- [Risk] 角色、工作流、审查插件未完整落地，容易被误解为本次都能执行 -> Mitigation: 在 spec 中明确“第一阶段执行插件类型”与“仅保留注册兼容”的边界。
- [Risk] 健康状态高度依赖宿主上报，断链时可能出现短暂不一致 -> Mitigation: 由注册中心显式标记 `degraded` 或 `unknown-like` 退化状态，而不是隐式假定插件仍可用。

## Migration Plan

1. 定义 manifest、插件记录和生命周期状态模型，作为后续代码和存储层的共同契约。
2. 在 Go 侧引入插件注册中心抽象，先支撑内置和本地来源的注册、查询、启停状态更新。
3. 在 TS Bridge 侧补齐工具插件运行时管理接口与状态上报，使 MCP 工具插件能纳入统一注册中心。
4. 在 Go 侧补齐集成插件运行时管理能力，并与注册中心共享统一状态机。
5. 最后把管理 API 或管理页面对接到注册中心，而不是直接读取宿主私有状态。

回滚策略：
- 如果双宿主运行时集成失败，可先保留注册中心与 manifest 校验层，把工具插件或集成插件其中一侧暂时标记为仅注册、不激活。
- 如果状态同步不稳定，可回退到“注册中心只展示静态元数据，不展示运行态”，但不撤销 manifest 与 capability 契约。
