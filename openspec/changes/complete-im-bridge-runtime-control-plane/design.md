## Context

当前 `src-im-bridge` 已经具备五个平台的单实例命令与通知能力，但运行方式仍然是“平台适配器直接收消息，后端通过 `/im/send`、`/im/notify` 直接回推”的轻量模式。这个实现足以覆盖 stub/live transport 验证，却还没有形成 PRD 和 `PLUGIN_SYSTEM_DESIGN.md` 描述的控制面闭环：Bridge 启动注册、实例级心跳、按 `bridge_id` 路由、断线重连后补发、长任务进度回传、以及对公网或内网入口的签名保护。

这次变更会同时触及 `src-im-bridge` 和 `src-go`。我们既要保留“单平台进程”的现有部署习惯，也要把 Bridge 升级成一个可被后端识别、可被安全调度、可恢复的运行时实例，而不是一个只能被动接 HTTP 请求的消息转发器。

## Goals / Non-Goals

**Goals:**
- 让每个 IM Bridge 实例以受控方式注册到后端，并持续报告平台、传输模式、能力、项目绑定和存活状态。
- 建立可恢复的 Bridge 控制面回传链路，支持通知、长任务进度和终态结果的定向投递、断线补发和重复投递抑制。
- 为长时命令补齐 IM 体验闭环：立即确认、运行中心跳、最终完成或失败摘要。
- 为现有 `/im/send`、`/im/notify` 等入口增加认证、签名和实例目标校验，避免未授权请求驱动真实 IM 平台。
- 让新增运行时模型和现有 `additional-im-platform-support` 能力兼容，不推翻单平台适配器的现有结构。

**Non-Goals:**
- 本次不把 IM Bridge 重写为完整的多平台同进程运行时；单进程单活动平台继续保留。
- 本次不一次性补齐 cc-connect 文档里全部 10 个平台适配器；范围聚焦当前已支持平台的运行时闭环。
- 本次不把整个 IM Bridge 直接迁移成最终版 Integration Plugin 形态；只把实例模型和控制面契约向该方向对齐。
- 本次不重做命令语义本身，例如 `/task`、`/agent`、`/review` 的业务规则。

## Decisions

### 1. Bridge 采用“显式注册的实例模型”，而不是隐式配置发现

每个 Bridge 进程在启动时生成或读取稳定的 `bridge_id`，并向后端注册以下信息：活动平台、传输模式、能力矩阵、回调入口、项目绑定、版本和启动时间。后端为实例维护 TTL/心跳状态，并在实例失活时停止向其投递新消息。

这样做的原因是，后续的定向通知、重复消费防护和断线恢复都需要一个可寻址的实例身份。仅靠部署配置或 `IM_PLATFORM` 无法区分多实例，也无法表达“这条消息应该回到哪个真实连接”。

备选方案：
- 只从 `/im/send`、`/im/notify` 的调用痕迹里反推活动 Bridge。被拒绝，因为无法保证实例唯一性，也无法表达离线/重连状态。
- 由运维在后端静态登记 Bridge 实例。被拒绝，因为扩缩容、重启和本地调试都会让静态登记很快失真。

### 2. 控制面使用“HTTP 注册 + 持久回传通道”的双层模型

Bridge 启动和关闭继续走显式 HTTP API（注册、心跳、注销），但通知、进度和终态结果的主回传路径改为实例级持久连接。设计上优先使用 WebSocket；如果需要兼容只读流，也允许后端保留 SSE 语义作为补充，但不把轮询作为主方案。

这样做的原因是：
- HTTP 适合幂等管理动作，不适合高频、低延迟、需要重连补发的事件投递。
- WebSocket 能同时承载通知、进度、ack、重放请求和心跳。
- 该模型与 PRD 中“WebSocket 接收通知推送”和插件系统中的事件流思路一致。

备选方案：
- 继续只保留 `/im/notify`、`/im/send` HTTP 推送。被拒绝，因为缺少重连恢复和实例在线感知。
- 全量改用 SSE。被拒绝，因为 SSE 单向，无法自然承载实例确认、补发游标和主动错误上报。

### 3. 回复上下文采用“后端可序列化存储 + Bridge 本地短期缓存”的混合策略

当 IM Bridge 把一条用户消息或命令归一化后，它必须把后续更新所需的最小回复上下文一起提交给后端，例如平台、会话键、聊天 ID、线程 ID、消息 ID、交互 token、延迟回复窗口等。后端以该上下文作为任务/运行实例和 IM 会话之间的关联依据；Bridge 侧只保留短 TTL 缓存，用于处理 provider SDK 不方便重复组装的对象。

这样做的原因是，Slack 线程、Discord interaction token、Feishu/Telegram chat context 的后续回复都依赖原始上下文；如果只在 Bridge 内存里保留，重连或实例切换后就无法可靠完成进度与结果回传。

备选方案：
- 完全只靠 Bridge 内存保存回复上下文。被拒绝，因为进程重启后会丢失，且不支持后端重放。
- 只把 task_id/project_id 发给后端，让后端自己猜目标会话。被拒绝，因为多数平台的后续回复需要显式的 provider 上下文。

### 4. 所有控制面入口都增加共享密钥签名和投递幂等键

现有 `/im/send`、`/im/notify` 以及新增的注册/心跳/回传接口都要求共享密钥或实例凭据；对后端到 Bridge 的投递增加签名头和 `delivery_id`，Bridge 维护最近处理的投递窗口来抑制重复发送。

这样做的原因是，当前裸 HTTP 入口一旦暴露给可访问网段，就可能被伪造请求直接驱动真实 IM 平台发消息。幂等键则用于解决重连、重放和重试导致的重复通知。

备选方案：
- 只依赖内网隔离或 `Authorization` 头。被拒绝，因为这不能覆盖跨进程投递重放和误配置场景。

### 5. 长任务交互统一为“三段式更新”，而不是只发最终结果

对 `SpawnAgent`、`QuickAgentRun`、任务分解、审查触发等长时操作，IM Bridge 在接收到命令后必须先给出即时确认；运行期间按状态变化和固定心跳节奏发送简短更新；结束时再发送终态摘要与后续入口（如任务、PR、审查结果链接）。

这样做的原因是，多数 IM 平台都有交互超时或用户可见性要求。只在最终完成时回消息会造成“命令像没生效”，而逐条镜像全部运行事件又会过于嘈杂并放大限频风险。

备选方案：
- 仅发送开始和结束两条消息。被拒绝，因为长任务期间完全不可见，无法满足 PRD 里的进度提醒与可追踪性目标。
- 直接把所有 Agent 输出逐条镜像到 IM。被拒绝，因为会触发平台限流，也会显著降低可读性。

## Risks / Trade-offs

- [控制面变复杂] → 先保留现有 HTTP 入口作为兼容层，分阶段切到注册 + 持久通道，避免一次切换所有部署面。
- [回复上下文跨平台差异大] → 统一定义可序列化的 `reply_target` 结构，并为 Slack/Discord/Feishu/Telegram 分别补充契约测试。
- [消息限流与心跳噪音] → 心跳只在长任务且状态无变化时按最小频率发送；平台支持编辑消息时优先更新原消息而不是新发一条。
- [多实例重复消费] → 后端投递显式携带 `bridge_id`，Bridge 侧再用 `delivery_id` 做本地幂等兜底。
- [迁移期行为分叉] → 在切流前要求兼容测试同时覆盖旧 HTTP 推送和新控制面事件通道，直到旧入口被明确降级为兼容模式。

## Migration Plan

1. 在后端建立 Bridge 注册表、心跳 TTL、签名校验和 `delivery_id` 基础设施，但暂时保留旧 `/im/send`、`/im/notify` 路径。
2. 让 `src-im-bridge` 在启动时注册并定时心跳，同时把健康信息、能力和项目绑定上报到后端。
3. 引入实例级持久回传通道，并让通知路径先双写到“旧 HTTP 推送 + 新事件通道”做灰度验证。
4. 为命令入口补充 `reply_target` 上报和长任务三段式更新；先从 `/agent run`、`/task decompose`、审查触发等明显长任务开始。
5. 当新通道稳定后，把旧裸 HTTP 入口降为受签名保护的兼容接口，仅供回滚和应急诊断使用。

回滚策略：
- 如果持久回传通道出现故障，后端可临时回退到已签名的 HTTP 推送兼容模式。
- 如果注册或心跳逻辑异常，Bridge 仍可按单实例本地模式启动，但后端不会把它视为可接收新投递的实例。

## Open Questions

- 第一版持久回传通道是否只实现 WebSocket，还是同时保留只读 SSE 供调试和诊断？
- `reply_target` 的权威存储放在数据库还是 Redis，更适合和任务/运行实例一起恢复？
- 第一版是否允许一个 Bridge 实例上报多个 `project_id` 绑定，还是先保持单 `AGENTFORGE_PROJECT_ID`，仅把协议设计成可扩展数组？
